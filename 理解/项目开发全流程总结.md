# ChartQA-X 三模型评测系统 — 开发全流程总结

> 本文档详细记录了从环境搭建到最终结论的完整过程，供撰写 20 页项目报告（实验章节、亮点与不足）参考。

---

## 一、项目目标

在 AMD ROCm 云平台上搭建 ChartQA-X 图表问答评测系统，对比三种技术路线：

| 模型 | 类型 | 输入 | 视觉→文本融合 |
|------|------|------|-------------|
| 规则基线 | 无视觉 | 问题文本 | ❌ 无融合 |
| LLaVA 1.5-7B | 外挂式多模态 | 图片 + 问题 | ViT → Projector → LLM |
| Gemma 4 E4B | 原生多模态 | 图片 + 问题 | Patch Embedding 直接生成软 Token |

## 二、环境搭建过程


### 2.3 遇到的版本问题及解决

| 问题 | 原因 | 解决 |
|------|------|------|
| `vllm==0.23.0+rocm723` 不存在 | wheels.vllm.ai 上该版本从未发布 | 不指定版本，让 uv 自动解析，最终 0.25.1+rocm723 |

## 三、模型部署与推理

### 3.1 关键发现：两个模型不能同时运行

48GB 显存无法同时加载 Gemma 4 E4B + LLaVA 1.5-7B。必须**串行**：
1. 启动 Gemma 4 服务（8000 端口）→ 运行推理 → Ctrl+C 停止
2. 启动 LLaVA 服务（8001 端口）→ 运行推理 → Ctrl+C 停止
3. 基线不需要 vLLM 服务，直接运行

### 3.2 必须加 `--served-model-name`

vLLM 启动时必须显式指定模型名，否则 batch_inference 的 API 调用会返回 404：

```bash
vllm serve /models/google/gemma-4-E4B-it/ --served-model-name gemma-4-E4B-it --port 8000
vllm serve /models/swift/llava-1.5-7b-hf --served-model-name llava-1.5-7b-hf --port 8001 --trust-remote-code
```

### 3.3 推理性能

- Gemma 4：2500 条约 50 分钟（1.3 秒/题）
- LLaVA：2500 条约 20 分钟（0.5 秒/题）
- 规则基线：2500 条约 1 秒（纯 CPU 计算）

## 四、评测代码的关键 Bug 修复

### 4.1 空串误判 Contains Match（最严重）

**现象**：基线 CM=50.4%，高于 LLaVA 11.2%，违反直觉。

**根因**：`normalize_answer("无法确定")` → 正则 `[^a-z0-9\s]` 去掉所有中文字符 → 空串 `""`。Python 中 `"" in "3"` 返回 `True`。

**影响**：基线 1105 条"无法确定"全部误判为匹配。真实 CM 仅 **6.2%**。

**修复**：`evaluator.py:46` 和 `error_analyzer.py:127`，标准化后若为空串则保留原文。

### 4.2 as_completed 乱序 bug

**现象**：跨模型交叉分析中 `g[i]` 和 `l[i]` 比较的是不同题目。

**根因**：`ThreadPoolExecutor` + `as_completed` 按完成顺序返回结果，非提交顺序。

**修复**：`batch_inference.py:177`，用 `indexed_results = {original_idx: result}` 按索引暂存，最后按序组装。

### 4.3 小数点被去除

**现象**：`normalize_answer("3.14")` → `"314"`

**修复**：`evaluator.py:47`，在去标点前用 `<DOT>` 占位保护数字间的小数点。

## 五、t-SNE 交融机制可视化

### 5.1 实现过程

1. 从 ChartQA-X 抽取 2500 样本
2. 对 LLaVA 和 Gemma 4 逐层提取 hidden states
3. 计算视觉/文本 Token 的平均向量 → t-SNE 降维
4. 计算跨模态余弦相似度随层数的变化曲线
5. 自动标注交融起始层

### 5.2 遇到的 Bug

| 问题 | 修复 |
|------|------|
| sklearn `n_iter` → `max_iter` | 参数名随 sklearn 1.2 变更 |
| 函数内 `import torch` 导致 NameError | 删除函数局部 import，用模块级 `torch` |
| `AutoModelForVision2Seq` 不存在于 transformers 5.x | 改用 `AutoModel.from_pretrained()` |
| LLaVA processor `images=` 参数顺序 | 必须放在 `text=` 前面 |
| Gemma 4 缺少 `<|image|>` token | 文本中显式加入 |
| torchvision ROCm 版无 PNG 支持 | 用 PIL 预加载图片传给 processor |
| 取样量被"简单/复杂"过滤截断 | 改为 `random.sample(data, max_samples)` |

### 5.3 最终结果

| 指标 | LLaVA | Gemma 4 |
|------|-------|---------|
| Layer 0 相似度 | 0.11 | 0.73 |
| 交融起始层 | Layer 24 | Layer 4 |
| Layer 32 相似度 | 0.66 | 0.99 |

**结论**：Gemma 4 从输入层就在同一空间，交融更早更彻底；LLaVA 需要 24 层才能对齐两个不兼容的向量空间。

## 七、最终实验结果

### 7.1 总指标（2500 题，修正后）

| 指标 | Gemma 4 | LLaVA | 基线 |
|------|---------|-------|------|
| Contains Match | **74.6%** | 11.2% | 6.2% |
| 精确数值全对 | **61.9%** | 2.8% | 0.4% |
| 视觉题 CM | **77.0%** | 15.9% | 2.2% |
| 平均回答长度 | 272 字 | 141 字 | 3 字 |

### 7.2 Venn 交叉分析

| 分类 | 题数 | 占比 |
|------|------|------|
| Gemma 4 独对 | 1521 | 60.8% |
| 三个都对 | 163 | 6.5% |
| 三个都错 | 321 | 12.8% |
| LLaVA 独对 | 36 | 1.4% |
| 基线独对 | 39 | 1.6% |

### 7.3 核心结论

1. **原生多模态（Gemma 4）显著优于外挂式（LLaVA）**：CM 6.7 倍，精确数值匹配 22 倍
2. **外挂式在数值读取上有根本性缺陷**：LLaVA 虽能理解图表语义，但 ViT→Projector 管道丢失了像素级精度
3. **交融机制差异是根因**：Gemma 4 从 Layer 0 就在统一空间，32 层全用于理解；LLaVA 前 24 层在"翻译"两个空间的差异
4. **规则基线仅靠问题文本撞库**：视觉题 CM 仅 2.2%，真正不看图时完全失败
