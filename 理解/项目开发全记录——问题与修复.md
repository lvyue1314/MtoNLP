# ChartQA-X 项目开发全记录

> 本文档记录了从零开始在 AMD ROCm 云平台搭建三模型评测系统过程中
> 遇到的全部关键问题及其解决方案，供撰写 20 页项目报告参考。

---

## 一、环境与部署阶段

### 1.1 vLLM 版本不存在

**问题**：`uv pip install 'vllm==0.23.0+rocm723'` 报 `No solution found`

**根因**：vLLM ROCm 预编译 wheel 的版本号格式取决于 ROCm 版本。云平台实际预装了 vLLM 0.16.1+rocm721，通过 `uv pip install vllm --extra-index-url https://wheels.vllm.ai/rocm/` 最终安装了 0.25.1+rocm723。覆盖安装自动升级了 transformers 4.57.6→5.14.1，使其能识别 Gemma 4 架构。

**教训**：不要在云平台上尝试安装一个"博客上写的"精确版本，先检查是否已预装，如需装则让 pip 自动解析。

### 1.2 两个模型不能同时启动

**问题**：48GB 显存无法同时运行 Gemma 4 E4B + LLaVA 1.5-7B 两个 vLLM 服务

**方案**：串行推理——先启 Gemma 4 服务（8000端口）→ 跑完评测 → Ctrl+C 停止 → 启 LLaVA 服务（8001端口）→ 跑完评测 → 停止。`main.py` 的 `check_services()` 已改为只检查用户请求的模型，不再要求两个服务同时在线。

### 1.3 gradio-client 版本冲突

**问题**：`gradio 4.44.0` 要求 `gradio-client==1.3.0`，但环境中有 `2.5.0`

**方案**：`uv pip install gradio --no-cache --reinstall` 让 uv 自动解析正确依赖。或者升级到 `gradio>=5.0`（原生支持 gradio-client>=2.0）。

### 1.4 PaddleOCR 3.x API 变更 + OneDNN 不兼容

**问题**：PaddleOCR 3.x 新增 PIR 编译器 + OneDNN 在 ROCm 上有 bug，导致 `use_gpu`、`show_log`、`cls` 参数报错，OCR 文本提取失败

**方案**：将基线从 OCR+LLM 改为纯规则匹配（从问题文本提取数字做启发式回答），不再依赖 PaddleOCR。误差分析的 OCR-aware 分类自动回退到纯文本规则。

## 二、测评代码 Bug

### 2.1 空串误判 CM（最严重的 bug）

**问题**：基线 50.4% Contains Match 远高于 LLaVA 11.2%，不符合直觉

**根因**：`normalize_answer("无法确定")` → 正则 `[^a-z0-9\s]` 去掉全部中文字符 → 空串 `""`。Python 中 `"" in "3"` 返回 `True`。1105 条"无法确定"全部误判为匹配。

**修复**：在 `evaluator.py` 和 `error_analyzer.py` 的 `normalize_answer` / `normalize_for_compare` 中添加空串检查——标准化后若为空则保留原始文本。

**影响**：基线真实 CM 从 50.4% 降至 **6.2%**。此前所有基于旧版 CM 的分析均不可用。

### 2.2 as_completed 乱序 bug

**问题**：`ThreadPoolExecutor` + `as_completed` 以**完成顺序**返回结果，而非提交顺序。2500 条推理结果随机排列。

**修复**：用 `dict {original_index: result}` 按原始索引暂存，最后按索引顺序组装。保存在 `batch_inference.py` 的 `run_inference` 方法中。

**影响**：LLaVA 11.2% 的 CM 是准确的（因为 CM 是行内匹配，与顺序无关），但此前 "Gemma 独对 824 题" 等跨模型交叉分析因为 `g[i]` 和 `l[i]` 比较的是不同题目而失真。修复后真正的 Gemma 独对是 **1521 题**。

### 2.3 小数点被去除

**问题**：`normalize_answer` 的 `re.sub(r"[^a-z0-9\s]", "", answer)` 会把 "3.14" 变成 "314"

**修复**：在去标点前先把数字间的小数点替换为 `<DOT>` 占位符，去标点后再还原。

## 三、t-SNE 交融分析

### 3.1 sklearn TSNE 参数名变更

**问题**：`TSNE.__init__() got an unexpected keyword argument 'n_iter'`——sklearn 1.2+ 把参数名改成了 `max_iter`

### 3.2 function-local import torch 导致 NameError

**问题**：函数内部 `import torch` + 函数开头使用 `torch.float16` → Python 把整个函数作用域的 `torch` 都标记为本地变量 → 之前对 `torch` 的引用报 "not associated with a value"

**修复**：删除函数内部的 `import torch`，使用模块级导入。

### 3.3 transformers 5.x 移除 AutoModelForVision2Seq

**问题**：Gemma 4 模型加载使用 `AutoModelForVision2Seq`，在 transformers 5.x 中已移除

**修复**：改用 `AutoModel.from_pretrained()` 自动检测架构。

### 3.4 LLaVA processor 图片参数顺序

**问题**：`processor(text=prompt, images=image)` 把文本当成图片 URL

**修复**：图片参数必须放在文本参数前：`processor(images=image, text=prompt)`

### 3.5 Gemma 4 processor 缺少 `<|image|>` token

**问题**：`processor(text=question, images=pil_img)` 报 "Found [0] <|image|> tokens and [1] images"

**修复**：文本中显式加入 `<|image|>` token：`processor(text="<|image|>\n" + question, images=pil_img)`

### 3.6 torchvision ROCm 版无 PNG 支持

**问题**：`decode_png: torchvision not compiled with libPNG support`

**修复**：用 PIL 预加载图片，传 PIL Image 对象给 processor，绕过 torchvision 解码。

### 3.7 t-SNE 取样量截断

**问题**：2500 样本只取到 1624 个——因为取样逻辑按"简单/复杂"分类取半，两类各约 800 条

**修复**：改为 `random.sample(data, max_samples)` 直接从全量随机取。

### 3.8 LLaVA 缺失 layer_32

**问题**：LLaVA 32 层，`hidden_states[layer_idx+1]` 对 layer_32 访问索引 33 越界

**原因**：无伤大雅——layer_28 的数据仍能说明高层融合情况。

## 四、数据集相关

### 4.1 列名大小写不匹配

**问题**：ChartQA-X 数据集实际列名为 `image_path`（PIL Image 对象）、`Question`（大写Q）、`Answer`（大写A），代码中用小写访问导致取不到数据

**修复**：改为 `sample["image_path"]`、`sample["Question"]`、`sample["Answer"]`

## 六、总结

整个项目中，代码逻辑 bug 和环境兼容性问题各占一半。最关键的教训：

1. **Python 空串匹配是常见陷阱**（`"" in any_string` 始终为 True）
2. **并发编程中顺序不可假设**（`as_completed` 不等于提交顺序）
3. **云平台依赖版本不透明**（先检查后安装，不盲信博客）
4. **多模态 API 碎片化严重**（同一个 processor 在不同版本中参数顺序、token 写法都可能不同）
