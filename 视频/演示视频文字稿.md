## 第 1 段：项目背景与数据集介绍（40 秒）

【画面】PPT 或直接展示图表文件：
- 三类模型架构对比表 → `理解/交融机制差异与测试差距的关联分析.md` 开头表格
- 数据集样例图片 → `data/chartqa_x/chartqa_images/LMUData/datasets/ChartQA/images/chart_000000.png`（任选3张柱状图/折线图/饼图）

```
大家好，我们做的项目是论文图表问答多模态评测系统。
全部实验在一台 AMD 的云平台上完成。

这台机器配置了 AMD 的高端 GPU，单卡就有 48GB 的显存，
系统内存高达 991GB，根目录磁盘总容量 3.5TB。
云平台提供了免费训练时长，我们无需本地 GPU，
通过网页就能申请实例、启动 GPU 环境、
用 vLLM 推理引擎部署 7B 级别的大模型。

数据集使用的是 ChartQA-X，这是一个专门为图表问答设计的
多模态数据集，由 HuggingFace 开源。它包含 2500 张不同的统计图表——
柱状图、折线图、饼图、堆叠图等，每张图配有自然语言问题和标准答案。
问题类型覆盖了数值读取、颜色识别、趋势分析、空间比较等多个维度。

我们在这个数据集上，对比了三种不同技术路线的表现：
第一条是纯规则基线——完全不看图，只从问题文本中提取数字做启发式回答；
第二条是 LLaVA 1.5-7B，代表外挂式多模态——它用一个独立的视觉编码器
   ViT 处理图片，再通过一个投影层映射到语言模型的文本空间；
第三条是 Gemma 4 E4B，代表原生多模态——它用 Patch Embedding
   直接把图像切成小块、投影到文本 Embedding 空间中。

我们想回答的核心问题是：
同样能"看图"，不同的融合方式，到底会导致多不一样的结果？
```

## 第 4 段：批量评测结果与误差分析（45 秒）

【画面】评测结果展示（按顺序切换以下文件）：
- 三模型对比表 → `结果+环境保存3/results/comparison/model_comparison.json`
- CM 柱状图 → `结果+环境保存3/results/comparison/model_comparison.png`
- 误差热力图 → `结果+环境保存3/results/comparison/error_matrix_fixed_heatmap.png`
- 误差饼图 → `结果+环境保存3/results/comparison/error_pie_charts.png`

```
我们在全部 2500 道题上跑了批量评测。

先说明一下评测指标的选择：
我们以 Contains Match 作为主指标——
它判断标准答案是否完整出现在模型的回答中。
为什么不选 Exact Match？因为多模态模型会自然地生成
详细的推理过程（Gemma 4 回答平均 272 字，
而标准答案平均只有 5 字）——精确匹配必然接近 0%。
Contains Match 在保留答案完整性的同时，
允许模型以自然语言形式输出。

最终结果：
Gemma 4 的 Contains Match 达到 74.6%，
LLaVA 只有 11.2%，
纯规则基线只有 6.2%。

我们还设计了精确数值匹配这个硬核指标——
直接提取 GT 中的所有数字，检查是否全部出现在回答中。
在这个指标上，Gemma 4 达到 61.9%，LLaVA 只有 2.8%——
差了整整 22 倍。

我们还对每个模型的失败案例做了四类错误分析。
看这张修正后的误差矩阵热力图：

Gemma 4 只失败了 635 道题，其中 503 道是幻觉——
模型看图后给出了一个数值，但这个数值在图表中并不存在。
说明即使是原生多模态，在精确数值读取上仍有局限。

LLaVA 的 2220 道失败题中，1738 道是幻觉——占比 78%。
它频繁编造数值：实际 3 条柱子说成 2 条，实际 62% 说成 75%。
这些错误恰恰验证了 ViT→Projector 管道在传输精确数值时的系统性失败。

基线只失败 2344 道，绝大部分被标为"文字识别错误"——
但这只是因为它的输出太简短，碰巧和编辑距离规则匹配。
实际上基线失败的核心原因是"根本不理解图表"。

三个模型的错误分布截然不同：
Gemma 4 的幻觉占比最高但总量少，
LLaVA 的幻觉泛滥成灾，
基线在视觉定位错误上为零——因为它根本没有视觉理解。
```

## 第 5 段：t-SNE 交融机制可视化——为什么 LLaVA 读不准数字？（50 秒）

【画面】t-SNE 可视化（按顺序切换）：
- t-SNE 矩阵图 → `tsne_visualization_v2/tsne_visualization/tsne_comparison_matrix.png`
- 跨模态相似度曲线 → `tsne_visualization_v2/tsne_visualization/cross_modal_similarity.png`

```
为什么同样是 7B 级别的多模态模型，
Gemma 4 和 LLaVA 的差距会达到 6.7 倍？

答案藏在模型内部的融合机制里。

我们做了 t-SNE 降维可视化——
对两个模型每层的视觉 Token 和文本 Token 提取 hidden states，
在 2D 空间中观察它们的分布关系。

蓝色点代表文本 Token，红色点代表视觉 Token。

先看 Gemma 4：
从第 0 层开始，蓝色和红色就已经混在一起了——
视觉和文本在输入层就被嵌入到了同一个向量空间。

再看 LLaVA：
第 0 层，红色和蓝色泾渭分明——完全分离。
第 12 层，依然分离。
一直要到第 24 层以后，它们才开始逐渐交汇。

这就是"外挂式"最根本的缺陷：
LLaVA 的 ViT 编码器输出的是视觉空间中的特征向量，
而语言模型工作在文本空间中。
它们生活在两套不同的坐标系里。

投影层（Projector）试图充当翻译者——
把一个 1024 维的视觉向量映射到 4096 维的文本空间。
但这个翻译是粗糙的、一次性的、有损的。
它只能做近似的对齐，无法精确传递每个像素的数值信息。

所以 LLaVA 的前 24 层 Transformer，
实际上都在做同一件事：
不断地拉近视觉特征向量和文本特征向量之间的语义距离——
它们像是一遍遍地去润色那个粗糙的翻译。
这些层的 Attention 计算不是在"理解图表"，
而是在反复地弥合两个向量空间之间无法消除的鸿沟。

而 Gemma 4，根本没有这个"翻译"步骤。
Patch Embedding 直接把图像块映射到文本 Embedding 空间——
从第 0 层开始，视觉 token 就是文本 token 的"同类"。
32 层 Attention，全部用于理解图表的内容。

这就是本质区别：
Gemma 4 有 32 层在理解，LLaVA 只有最后 8 层。
一个全力输出，一个打了四分之一折。
```

【画面】跨模态相似度曲线特写 → `tsne_visualization_v2/tsne_visualization/cross_modal_similarity.png`

```
这张曲线图更直观地呈现了这个差距。

横轴是层数，纵轴是视觉和文本的余弦相似度。

Gemma 4 的线条，从第 0 层就高达 0.73，
到第 4 层就开启了"交融加速"模式，一路平滑上升到最终的 0.99。

LLaVA 的线条，从 0.11 的极低点起步，
在前 24 层几乎以恒定速率缓慢爬升，
在相似度曲线上没有任何显著的"融合拐点"——
因为所有层都在做的不是融合，是翻译。

两条曲线的对比，把"原生统一空间"和"外挂式投影对齐"的差距，
变成了一道可视的数学证明。
```

## 第 6 段：规则基线失败分析 + 总结（30 秒）

【画面】回到 PPT：三类模型对比 + 典型案例

```
最后回到规则基线——
它在视觉依赖题上只有 2.2% 的 CM，证明了一个朴素的道理：
不看图，就不可能回答需要看图的问题。

但这正是我们设计这个对照实验的目的。
我们不是要证明 Gemma 4 "很强"——
而是要证明：多模态模型的关键差异，
不在于参数量，不在于训练数据，
而在于视觉和语言信息的融合方式。

Patch Embedding 让视觉从"外来者"变成"自己人"，
32 层 Transformer 全部投入到理解任务中，
这是原生多模态的正确方向。

反之，外挂式的 ViT→Projector→LLM 架构，
在需要像素级精度的图表问答上，
存在一个无法通过扩大模型规模来解决的结构性瓶颈。

我们的全部代码、评测数据和可视化结果
都已经开源在 GitHub 上，欢迎各位查看和复现。

谢谢大家。
```
